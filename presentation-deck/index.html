Problem

No reason for closers to be on the platform.

68% of SalesRabbit's revenue comes from roofing, solar, home improvement, and security. These industries run on setter-closer teams. But closers had no reason to pay for a seat. Between no Round Robin, limited calendar integration, and broken lead access, the scheduling system just wasn't built for how these teams work. Every team we onboarded either churned or switched to third-party tools.

Iteration

Rejected, then refined

Early concepts restructured the Set Appointment flow with a new layout. Internal users rejected it. The UI looked too different from the existing lead detail, so users weren't sure if the fields were capturing the same information or something separate.

The learning

Familiarity builds trust. The final design maintained the lead detail's structure and patterns while adding scheduling-specific sections. Users immediately understood what each field did.

Reflection

What this project taught me

Setup is a design problem, not an implementation detail.

The scheduling UX worked. Users who got through setup used it confidently. But setup required help every time. I treated onboarding as something engineering would figure out. It was the most important design problem and I designed it last.

The dispatch model broke our assumptions.

We designed for round-robin. The most common real-world pattern was dispatch: one manager receiving all appointments and redistributing manually. We didn't see it until early access. Earlier field research would have caught it.

Recommending a delay is a design decision.

The feature was technically ready. I recommended we hold the launch because setup would create a support crisis. That call came from the research, not from product or engineering. Knowing when not to ship is part of the job.

The Problem

The translation gap

Between describing a design idea and seeing it rendered, there are too many translation steps. A designer says "give me a hero with a bold heading and a warm CTA," then opens Figma, drags frames, sets tokens, exports, writes a ticket. The idea loses fidelity at every handoff.

I wanted to explore: what if speech was the input and working UI was the output?
