How Design Capture works

Talk through your ideas. AI listens, extracts, generates, and you triage. Six steps from conversation to components.

1·Human
Designer speaks
Hit record and describe what you want. The Web Speech API transcribes in real time. The system extracts ideas from your stream of consciousness.

Summary
Who does what
Designer
Talks through ideas. Reviews output. Refines with plain language. Directs — doesn't code.
Design System
The constraint layer. Wraps every generation with tokens. Makes AI output safe to ship.
Claude AI
Extracts ideas. Generates HTML. Applies refinements. Executes — doesn't decide.
The insight
Voice is the most natural design tool. Designers already talk through ideas in reviews, crits, and standups. Design Capture turns that conversation into components — no translation step, no ticket, no handoff.

Findings

Problems encountered while building the voice-to-UI pipeline, and how each was resolved. A living log of prompt engineering, rendering, and extraction challenges.

1
Sonnet labels components wrong
User says "badge" but extraction returns "button" as componentType
Extraction
Problem
The extraction prompt listed generic component types (hero, nav, card, form, etc.) as examples. Sonnet would map "badge" to the nearest known type — "button." Even when the user explicitly said "badge," the model defaulted to familiar categories.
Solution
Rewrote the extraction prompt with an exhaustive componentType list (badge, tooltip, tag, avatar, chip, pill, etc.) and added the rule: "Use the EXACT element name the speaker used." Added 3 few-shot examples showing correct extraction, including one where "badge" stays "badge."
componentType: "the specific element type the speaker mentioned — use their exact word: badge, tooltip, tag, avatar, icon, pill, chip, toggle, switch, dropdown..." // Few-shot example: Transcript: "add a badge to the AI Chat button" Output: { componentType: "button", actionType: "modify" }
2
Modifications extracted as new components
"Fix the text wrapping" creates a standalone label instead of modifying the existing one
Extraction
Problem
The extraction prompt had no concept of the page's current state. It couldn't distinguish "fix something that exists" from "build something new." Every idea came back as actionType: "create" with no reference to existing elements.
Solution
Three changes: (1) Added getPageContentSummary() that scrapes all data-component attributes from the page iframe and includes them in the prompt. (2) Added targetElement field mapped to data-component names. (3) Made actionType default to "modify" with the rule: "adding something TO an existing element = modify that element." Few-shot examples show the distinction clearly.
// Page context injected into extraction prompt: [Hero Label] <p> text: "Product Designer · AI Engineer · Salt Lake City" tokens: font: JetBrains Mono 12px/400, color: #F4B56F [Hero Heading] <h1> text: "Designing" tokens: font: Outfit clamp(80px,10vw,140px)/700
3
Generated badges aren't round
Sonnet creates square or rectangular badges instead of circular dots
Generation
Problem
Telling Sonnet to "add a badge dot" produced various shapes — pills, squares, rounded rectangles. The design system's 4px border-radius was being applied to everything, including elements that should be circles.
Solution
Added an explicit BADGE PATTERN block to buildGenPrompt() that fires when the description contains "badge," "dot," or "indicator." It provides the exact CSS as a copy-paste snippet rather than trusting Sonnet to infer the visual.
// Explicit CSS in prompt: Add a child: <span style="position:absolute; top:-3px; right:-3px; width:10px; height:10px; border-radius:50%; background:#e8461e; display:block;"></span>
Takeaway
Sonnet needs exact CSS values for visual patterns. Vague instructions like "circular badge" fail ~40% of the time. Providing the literal style string brings it to ~95%.
4
Previews show elements in isolation
A text wrap fix looks correct but has no surrounding context to judge against
Generation
Problem
When modifying a single element (like the hero label), the generation prompt only included that element's HTML. The preview showed just a line of gold text on a dark background — no heading, no nav, nothing to compare against. You couldn't tell if the fix actually worked.
Solution
Added getElementContext() that walks up the DOM to find the nearest parent with a data-component attribute. The generation prompt now includes both the target element AND surrounding context, with instructions to "include neighboring elements so the change is visible in context."
// getElementContext() walks up to parent component: Target: Hero Label → Parent: Hero Section Prompt includes the full hero section HTML so the preview shows: label + heading + CTA together
5
Mobile fix shows the problem, not the solution
Text wrap fix renders at desktop width where wrapping doesn't occur
Generation
Problem
The prompt said "fix for mobile" but the generated HTML rendered at whatever width the capture iframe was — usually wide enough that no wrapping occurred. Or it wrapped in a 375px div but didn't actually change the CSS, just showing the same broken layout at mobile width.
Solution
Rewrote the mobile branch of buildGenPrompt() to be explicit: "Wrap in width:375px. APPLY THE FIX — show text fitting properly, not still broken. Fix options: reduce font-size, add white-space:nowrap, flex-wrap." The key phrase: "The output must show the SOLUTION."
// Before: "Show the element AS IT WOULD APPEAR AT 375px" → Sonnet just shows the broken version at 375px // After: "APPLY THE FIX — show text fitting properly" → Sonnet actually modifies the CSS to fix it
6
Large elements render as white blobs
Hero heading at 140px overflows the capture frame, producing a white circle
Rendering
Problem
The hero heading uses font-size: clamp(80px, 10vw, 140px). When "make it bigger" increased this further, the rendered text exceeded the 500×300 max capture dimensions. html2canvas clipped it or produced artifacts — typically a white circle or blob in the thumbnail.
Solution
Updated renderAndCapture() to detect oversized content and apply CSS transform: scale() to shrink it proportionally before capture. Max bounds increased to 600×400. If content exceeds that, a scale factor is computed and applied to transform-origin: top left.
// Auto-scale for oversized content: if (rawW > maxW || rawH > maxH) { const scale = Math.min(maxW/rawW, maxH/rawH); d.body.style.transform = `scale(${scale})`; d.body.style.transformOrigin = 'top left'; }
7
Thumbnails tiny or off-center
Small components (badges, buttons) appeared as dots in the upper corner of thumbnails
Rendering
Problem
The capture iframe was fixed at 800×600 with the component rendered at actual size. A 120px button appeared as a tiny dot in the corner of a huge dark canvas. The thumbnail's object-fit: contain shrank it further.
Solution
Changed the render iframe body to display:flex; align-items:center; justify-content:center. Added auto-sizing: measure the component's bounding box, set canvas to content size + 32px padding. Changed card thumbnails to max-height:100px; object-fit:cover; object-position:center.
8
Header borders don't align across panels
Toolbar, column headers, and transcript panel had competing border lines at different heights
Layout
Problem
Three horizontal headers competed: the tasks toolbar (padding-based height), the kanban column headers (own border-bottom), and the transcript panel header (different padding). Each had slightly different heights, creating misaligned horizontal lines.
Solution
Set all three headers to a fixed height: 45px with padding: 0 vertical. Removed the column header border-bottom entirely — columns are separated by vertical borders only, with the single toolbar border serving as the horizontal divider.
9
In-iframe toolbar interferes with voice recording
Edit/Delete buttons in the page iframe captured clicks intended for element selection
Layout
Problem
A floating Edit/Delete toolbar appeared inside the page preview iframe when elements were selected. During voice recording, the user was selecting elements to reference them in speech — but the toolbar intercepted clicks and messed up selections.
Solution
Removed the in-iframe toolbar entirely. Moved Edit Text and Delete Element buttons to the Token Editor drawer (right side panel), which opens on element selection. The iframe now only handles selection highlighting — editing actions happen in the drawer via postMessage.
10
Icon swap doesn't enable "Add Option"
Changing an icon in the Token Editor doesn't register as a pending change
Layout
Problem
The "Add Option" button in the Token Editor was gated by Object.keys(tePendingChanges).length > 0. CSS changes (font-size, color, padding) populated tePendingChanges, but icon swaps via stageIcon() only sent a postMessage to the iframe — they didn't update the pending changes object.
Solution
Added tePendingChanges._icon = name and buildCreateActions() call to stageIcon(). Now icon changes register as pending, enabling the Add Option button.
