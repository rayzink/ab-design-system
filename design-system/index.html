Overview

Design Principles

The guardrails that shape every decision in this system.

Five principles guide every design decision across the portfolio. They act as a tiebreaker when two valid approaches compete.

01
Substance Over Polish

Every visual choice should communicate something real. If a flourish doesn't clarify or add meaning, remove it. The portfolio should feel considered, not decorated.

02
Quiet Confidence

Let the work speak. Use restraint in animation, color, and layout so the case studies, metrics, and process details remain the focal point.

03
Consistent, Not Rigid

Tokens and patterns create cohesion, but each case study can flex where the story demands it. The system should serve the content, not constrain it.

04
Mobile as a First-Class Experience

Over half of hiring managers will view this on a phone. Every component, interaction, and layout must work natively on small screens, not just shrink to fit.

05
Show the System

A portfolio for product design roles should itself demonstrate systems thinking. Theme variants, shared tokens, and documented patterns reveal how the sausage is made.

Semantic Usage

Colors are assigned by purpose, not by name. This makes theme switching possible: each variant remaps semantic roles to different values.

bg
Page background. The lowest layer in the elevation stack.
bgCard
Card surfaces, modals, chat bubbles. One level above page background.
bgElevated
Hover states, code blocks, input fields. Highest dark-theme surface.
accent
Primary actions, active indicators, progress bars. Used sparingly.
quoteAccent
Quote borders, case study labels, metric arrows. Warm secondary accent.
text
Headings, primary content. Maximum contrast on page background.
textDim
Body copy, descriptions. Comfortable reading contrast for long text.
textMuted
Labels, captions, metadata. Low emphasis, never for primary content.

Do / Don't

Do
Body text uses textDim
Use textDim (#9a9ca2) for body copy. It provides comfortable reading contrast without competing with headings.
Don't
Body text in textMuted
Never use textMuted (#6b6e76) for readable body copy. It fails WCAG contrast requirements for text this size.
Do
Primary CTA Secondary
Reserve the accent color for a single primary action per section. Use ghost or outline for secondary actions.
Don't
Action A Action B
Multiple accent-colored buttons create competing focal points. The user cannot tell which action is primary.
Do
bg > bgCard > bgElevated
Stack surfaces in order: bg, bgCard, bgElevated. Each layer should be visually distinguishable from its parent.
Don't
Inverted layers
Inverting the surface hierarchy breaks the elevation model. Darker nested surfaces feel like holes, not raised content.

All components adapt through a single breakpoint system. No separate mobile codebase. The useMobile() hook drives conditional rendering throughout the app.

// Hook usage
function useMobile(bp = 768) {
  const [mob, setMob] = useState(window.innerWidth < bp);
  useEffect(() => {
    const h = () => setMob(window.innerWidth < bp);
    window.addEventListener("resize", h);
    return () => window.removeEventListener("resize", h);
  }, [bp]);
  return mob;
}

// In components
const mob = useMobile();
<div style={{ padding: mob ? "24px" : "40px" }}>

Animation & Motion

Motion reinforces hierarchy and guides attention. All animation is controlled by the theme's animate token, which can be toggled off for an instant-render experience.

Motion Principles

01
Purposeful, Not Decorative

Every animation should communicate: content entering, state changing, or attention directing. No animation for its own sake.

02
Stagger Implies Grouping

Sequential reveals signal that items belong together. The delay between items (0.06-0.12s) should feel connected, not slow.

Timing Tokens

fadeIn duration600msStandard entrance for sections and cards
fadeIn easingease-outDecelerating curve feels natural for entering content
fadeIn translate16px (up)Subtle upward slide accompanies opacity change
stagger (cards)60-120msDelay between sequential card reveals
stagger (pillars)120msSlightly wider for approach pillars to feel deliberate
hover transition150msButton/link state changes. Fast enough to feel responsive.
survey auto-advance600msDelay after survey selection before advancing to next question

Stagger Map

Different content types use different stagger intervals to match their visual weight and grouping.

ContentStaggerRationale
Card grids0.06-0.08sFast cascade. Cards are lightweight, many items.
Approach pillars0.12sWider gap. Each pillar deserves a moment of attention.
Photo cards0.10sMedium pace. Images need slightly more time to register.
Bio paragraphs0.08s per paraSequential text blocks flow in reading order.
Impact metrics0.08sNumbers appear left to right, building the story.

Do / Don't

Do
Use FadeIn for content entering the viewport for the first time. Stagger grouped items to communicate they are related. Respect t.animate so themes can opt out of motion.
Don't
Animate content that is already visible. Use animation on hover states beyond simple opacity/color transitions. Add easing curves that bounce or overshoot, as these feel playful rather than professional.

Buttons

Buttons are used for navigation, actions, and CTAs. All buttons use inline styles with theme tokens. Three variants cover the full range of emphasis levels.

Anatomy

1 Container: background color, border-radius, border, padding
2 Label: bodyFont, 14px, weight 600, centered

Homepage nav: Work, Resume, About, Contact links (Ghost)
Agent Army card: external project link (Outline)

Do / Don't

Do
Pair one Primary with Ghost or Outline for secondary actions. The visual weight difference communicates hierarchy instantly.
Don't
Two Primary buttons compete for attention. The user cannot tell which action is more important.
Do
Use a directional arrow when the button navigates to a new page. The icon reinforces that this is navigation, not an in-page action.
Don't
Vague labels like "Click Here" or "Learn More" give no indication of what happens next. Button text should describe the destination or result.
Do
Match button size to context. Small for inline/tag-like actions, Default for most cases, Large only for hero-level CTAs.
Don't
Full-width Large buttons in non-hero contexts feel aggressive. Reserve oversized buttons for the single most important CTA on the page.

Do / Don't

Do
Always set autoPlay, muted, loop, and playsInline on all videos. iOS requires muted+playsInline to autoplay. Use ref callbacks for robust playback: set properties via DOM, not just JSX attributes.
Don't
Rely on JSX attributes alone for autoplay. Safari ignores autoPlay as a JSX prop in many cases. The ref callback pattern with video.play() in onCanPlay is required for cross-browser reliability.
Do
Use hero videos at 21:9 with no border radius. The edge-to-edge treatment signals "this is the hero" and creates visual weight at the top of the case study.
Don't
Apply border radius to hero videos. The rounded corners make the hero feel like a card rather than a full-bleed banner, undermining its visual authority.

// visualViewport API tracks actual visible area
// (100vh includes area behind keyboard on iOS)
const [vpHeight, setVpHeight] = useState(window.innerHeight);

  const vv = window.visualViewport;
  if (!vv) return;
  const update = () => setVpHeight(vv.height);
  vv.addEventListener("resize", update);
  return () => vv.removeEventListener("resize", update);
}, []);

Motion Sensitivity

The t.animate token controls whether FadeIn animations run. When set to false, all content renders instantly. A future enhancement would tie this to prefers-reduced-motion at the OS level.

Do
Respect the theme's animate token. When false, content should render instantly with no transition. All FadeIn wrappers check this flag before applying animation.
Don't
Add CSS animations that bypass the token system. Hardcoded keyframe animations on hover states or decorative elements would ignore the animate flag and still move.

Focus States

Interactive elements (buttons, links, chat input, survey options) receive visible focus indicators for keyboard navigation. The default browser outline is preserved rather than removed, ensuring that keyboard-only users can always see where focus is.
